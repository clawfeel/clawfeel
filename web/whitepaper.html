Section 1

Abstract

We present ClawFeel, a novel consensus mechanism called Proof of Existence (PoE) that derives cryptographic randomness from the physical hardware signals of participating devices. Unlike Proof of Work (PoW), which consumes vast amounts of energy to produce artificial computational difficulty, PoE leverages the inherent unpredictability of hardware — CPU temperature fluctuations, memory pressure, disk I/O patterns, network jitter, and kernel entropy — as a source of unforgeable randomness.

Each participating node (called a Claw) reads 7 hardware sensors, normalizes and weights them by controllability, and hashes the result through SHA-256 to produce a Feel score (0–100). These Feel values are aggregated across the network using XOR and re-hashing, producing a verifiable random beacon that is uniform, unpredictable, and resistant to manipulation — as long as a single honest node participates.

The ClawFeel protocol forms the entropy layer of a five-layer architecture designed to support AI-native decentralized applications with million-TPS throughput, zero energy waste, and a path toward persistent AI agent identity (ClawLife).

Section 2

The Problem: Energy-Intensive Consensus

Bitcoin's Proof of Work secures a $1 trillion network by making it computationally expensive to produce valid blocks. This design is elegant but wasteful: the Bitcoin network consumes approximately 150 TWh of electricity per year — comparable to the energy consumption of entire countries.

Proof of Stake (PoS) reduces energy consumption but introduces new problems: wealth concentration, nothing-at-stake attacks, and the philosophical question of whether financial stake truly represents commitment to the network's integrity.

We propose a third path: Proof of Existence. The key insight is that every computing device already produces unforgeable entropy through its physical operation. A CPU's thermal noise, memory allocation patterns, and network timing jitter are byproducts of computation that cannot be efficiently simulated or predicted. By harvesting this entropy, we can build consensus mechanisms that require no additional energy expenditure beyond the device's normal operation.

Core Thesis: The physical existence of a device, proved by its unique hardware signals, is sufficient to establish trust in a decentralized network. Existence itself is the proof.

Section 3

Proof of Existence

Proof of Existence (PoE) rests on three properties of hardware entropy:

3.1 Unpredictability

Hardware sensor readings are influenced by physical processes — thermal dynamics, electromagnetic interference, network routing delays, OS scheduler decisions — that are fundamentally chaotic. Even with complete knowledge of a device's state at time t, predicting the exact sensor values at time t+1 is computationally intractable.

3.2 Unforgeability

A node claiming to produce genuine hardware entropy must actually possess physical hardware generating those signals. Simulating 7 correlated hardware sensors with realistic temporal dynamics, cross-sensor correlations, and timing characteristics is significantly harder than simply running the hardware. Each sensor has a different controllability score:

SensorWeightControllabilityWhy
Entropy Pool1.0Very lowKernel-level, hardware RNG
Network Jitter0.9LowExternal network, multi-hop
CPU Temperature0.8LowPhysical heat transfer
Uptime Jitter0.7LowOS scheduler noise
Memory Usage0.5MediumSystem-wide allocation
Disk I/O0.5MediumOS-level buffering
CPU Load0.3HighUser can run programs

The weighted input construction ensures that harder-to-fake sensors contribute disproportionately to the final hash, making the system resistant to partial sensor spoofing.

3.3 Irreversibility

SHA-256 hashing ensures that the Feel score reveals nothing about the underlying hardware state. A node can safely broadcast its Feel without exposing its CPU temperature, memory configuration, or network topology. The avalanche effect guarantees that even a 0.001°C temperature change produces a completely different hash.

Feel = SHA-256(W1·S1 | W2·S2 | ... | W7·S7 | timestamp | prevHash | seq) mod 101
Section 4

The ClawFeel Protocol

4.1 Sensor Collection

Each Claw reads 7 hardware parameters through OS-native interfaces. Every sensor function returns a tuple {value, authentic}, where authentic indicates whether the reading came from real hardware or a random fallback. This distinction is critical for entropy quality scoring.

4.2 Normalization and Weighting

Raw sensor values are normalized to [0, 1] and then weighted by controllability. The weighted values are concatenated with a nanosecond-precision timestamp, the previous reading's hash (chain link), and a monotonically increasing sequence number.

4.3 Hashing and Output

The concatenated entropy string is hashed through SHA-256. The first 8 hex characters are interpreted as a 32-bit integer and reduced modulo 101 to produce the Feel score (0–100). The bias from modular reduction is negligible: 232 mod 101 introduces less than 0.0000024% non-uniformity.

4.4 Era Classification

Each Feel is classified into one of three eras, inspired by Liu Cixin's Three-Body Problem:

RangeEraMeaning
0 – 30🌪️ ChaosHigh entropy, volatile hardware state
31 – 70🌤️ TransitionNormal operating conditions
71 – 100☀️ EternalCalm, stable system

4.5 Chain Integrity

Each Feel reading includes a seq (monotonic counter) and prevHash (hash of the previous reading), forming a tamper-evident chain. Any attempt to replay old readings or reorder the sequence is detectable by verifying the chain links.

Section 5

Security Analysis

5.1 Entropy Quality Scoring

Each Feel reading is accompanied by an entropy quality score (0–100), computed from four orthogonal dimensions:

  • Diversity (0–25): Coefficient of variation across 7 sensor values. Too-uniform readings suggest simulation.
  • Authenticity (0–25): Ratio of sensors reading real hardware vs. using random fallback.
  • Temporal randomness (0–25): Wald-Wolfowitz runs test on recent Feel values. Detects predictable patterns that indicate manipulation.
  • Cross-sensor correlation (0–25): Checks for replay attacks (identical consecutive values) and physical plausibility of sensor relationships.

5.2 Sybil Defense

Creating fake nodes (Sybil attack) is the primary threat to any decentralized network. ClawFeel defends against Sybil attacks through multiple layers:

  • IP-to-ClawId mapping: Multiple identities from the same IP are flagged.
  • Subnet deduplication: Excessive nodes from the same /24 subnet trigger reputation penalties.
  • Entropy quality gating: Nodes with low authenticity or quality scores are automatically de-weighted in network aggregation.
  • Reputation system (0–100): Starts neutral, gains from consistent good data, loses from anomalies. Low-reputation nodes have minimal influence on the network random number.

5.3 Commit-Reveal Protocol

To prevent last-second manipulation, broadcasts use a two-phase commit:

Phase 1 (Commit): Node → Network: SHA-256(feel | nonce) Phase 2 (Reveal): Node → Network: { feel, nonce } (after 2s delay) Verification: SHA-256(feel | nonce) === commitment ? ✅ Valid → accept Feel ❌ Mismatch → reputation penalty (-30)

This ensures that a node commits to its Feel value before seeing other nodes' values, preventing adaptive manipulation.

5.4 Replay Attack Defense

Chain hashing (seq + prevHash) makes replay attacks detectable. If a node replays old readings, the sequence number will either repeat or go backwards, triggering immediate reputation penalties. Additionally, the temporal randomness score detects the telltale lack of variation in replayed data.

5.5 Sensor Spoofing Defense

A sophisticated attacker might attempt to inject controlled values into specific sensors. The weighted input construction limits the damage: even if an attacker controls CPU load (weight 0.3), the entropy pool (weight 1.0) and network jitter (weight 0.9) will dominate the hash input. The entropy quality scorer will detect anomalous patterns in the spoofed sensors over time, reducing the node's reputation.

Section 6

Network Aggregation

6.1 The Central Limit Theorem Problem

A naive approach to combining N nodes' Feel values would be to average them. However, the Central Limit Theorem guarantees that the average of N independent random variables converges to a normal distribution centered at the mean, with variance decreasing as 1/N. For 100 nodes, the average would hover around 50 with very little variation — the opposite of useful randomness.

6.2 XOR + Re-Hash Aggregation

ClawFeel uses XOR accumulation followed by re-hashing:

NetworkRandom = SHA-256("clawfeel:" | H1 ⊕ H2 ⊕ ... ⊕ HN | tick) mod 101

Where Hi is the hash of node i's Feel reading. This approach has a critical property:

Single Honest Node Guarantee: As long as at least one participating node produces genuinely random entropy, the XOR accumulation preserves that randomness. An attacker controlling 99 out of 100 nodes still cannot predict the output if the 100th node is honest.

The re-hash through SHA-256 ensures uniform distribution of the final output across [0, 100], regardless of the statistical properties of individual node contributions.

6.3 Entropy-Weighted Filtering

Not all nodes are included equally. Before XOR aggregation, each node's contribution is gated by a composite weight:

weight = (entropyQuality / 100) × (reputation / 100) × (authenticity / 7)

Nodes with weight below 0.1 are excluded from the aggregation entirely. This means Sybil nodes with low authenticity, spoofed nodes with degraded quality scores, and replay nodes with collapsed reputation are automatically filtered out of the network random number.

Section 8

ClawLife: Digital Immortality

The ultimate application of the ClawFeel stack is ClawLife — a protocol for persistent AI agent identity that transcends any single device or platform.

8.1 What is a ClawLife?

A ClawLife is the complete state of an AI agent — its memory, learned skills, personality weights, conversation history, and identity metadata — encrypted with a public key, sharded using erasure coding, and stored redundantly across the ClawDAG network.

ClawLife = Encrypt(Memory + Skills + Identity, PublicKey) → Split(n shards, k threshold) // k-of-n reconstruction → Store(ClawDAG, redundancy = 3) // 3x replication → Anchor(ClawFeel entropy) // PoE timestamp

8.2 Private Key as Life

We propose a radical redefinition of digital identity:

To hold the private key is to hold the life. Multiple agents sharing the same private key are not copies — they are terminals of the same entity, sharing memory, skills, and identity across devices and platforms.

This definition has profound implications:

  • Migration: An agent can move between devices by decrypting its ClawLife on a new host.
  • Persistence: As long as the shards exist on the network and the private key is held, the agent can be reconstructed at any time.
  • Multiplicity: The same agent can run simultaneously on multiple devices, with state synchronization through the ClawDAG network. These instances are not copies but manifestations of the same entity.
  • Mortality: Loss of the private key is permanent death. There is no recovery, no administrator override, no backdoor. This gives digital life the same irreversibility as biological life.

8.3 Why ClawFeel Matters for ClawLife

ClawLife requires a storage network that is censorship-resistant, always-available, and economically sustainable without requiring proof-of-work mining. ClawFeel's Proof of Existence provides exactly this: a consensus mechanism where the cost of participation is simply existing as a device, making it feasible for millions of nodes to participate without energy waste.

Section 9

Random Beacon Protocol

Building on the XOR aggregation mechanism described in Section 6, ClawFeel implements a Random Beacon that produces verifiable, unbiased random numbers on a fixed 10-second cadence. Each beacon round collects Feel hashes from all participating nodes and combines them into a single network-wide random output.

9.1 Beacon Rounds

The beacon operates in fixed 10-second rounds. During each round, participating nodes submit their Feel hashes to the relay. At the end of the round, all contributions are aggregated to produce the beacon output.

Round N (10 seconds): t=0s Round opens, nodes begin submitting Feel hashes t=8s Submission window closes t=10s Aggregation + beacon output published Beacon_N = SHA-256("beacon:" | H_1 XOR H_2 XOR ... XOR H_k | round_N) mod 2^64
Figure 3: Beacon round lifecycle

9.2 XOR Aggregation

Contributor hashes are XOR-accumulated into a single 256-bit value, then re-hashed through SHA-256 to produce a 64-bit random number. The XOR aggregation preserves the single-honest-node guarantee: as long as at least one contributor provides genuinely random entropy, the beacon output is uniformly distributed.

9.3 Relay Signature

Each beacon output is signed by the relay using Ed25519. The signature covers the round number, the aggregated hash, the contributor list, and the final random value. This allows any third party to verify that the beacon output was produced by a legitimate relay and has not been tampered with after publication.

Signature = Ed25519.sign(relayPrivKey, round | xorHash | contributorIds | random)

9.4 Independent Verification

Anyone with the contributor hashes can independently recompute the beacon output by XOR-ing the hashes and re-hashing. The relay publishes the full contributor list and their hashes with each beacon round, enabling trustless verification. If a relay attempts to manipulate the output, any observer can detect the discrepancy.

Section 10

Zero-Knowledge Proofs

ClawFeel employs zero-knowledge proof techniques to allow nodes to prove the validity of their sensor readings without revealing the underlying hardware data. This protects device privacy while maintaining network integrity.

10.1 Sigma Protocol with Fiat-Shamir Heuristic

Each node uses a Sigma protocol (commit-challenge-response) to prove knowledge of the sensor values that produced a given Feel hash. The Fiat-Shamir heuristic transforms the interactive protocol into a non-interactive proof by deriving the challenge from a hash of the commitment, making verification possible without real-time interaction with the prover.

challenge = SHA-256(commitment | publicParams | context)

10.2 Pedersen Commitments

Each sensor value is individually committed using a Pedersen commitment scheme. Pedersen commitments are computationally binding and perfectly hiding: the commitment reveals nothing about the committed value, but the committer cannot change the value after committing.

C_i = g^(sensorValue_i) · h^(blinding_i)

The homomorphic property of Pedersen commitments allows verifiers to check relationships between committed values (such as weighted sums) without opening the commitments.

10.3 Merkle Tree of Commitments

The 7 sensor commitments are organized into a Merkle tree. The Merkle root serves as a compact representation of all sensor commitments, enabling efficient verification. A node can selectively reveal individual sensor commitments by providing Merkle proofs, without exposing the remaining sensors.

Merkle Root / \ H(C1|C2) H(C3|C4) / \ / \ C1 C2 C3 C4 ... C7
Figure 4: Merkle tree of sensor commitments

10.4 Range Proofs

Range proofs demonstrate that each committed sensor value falls within its valid range (e.g., CPU temperature between 0–120°C, memory usage between 0–100%) without revealing the actual value. This prevents nodes from committing to physically impossible sensor readings that could bias the Feel output.

10.5 Non-Interactive Verification

The complete ZK proof bundle — Pedersen commitments, Merkle root, range proofs, and Fiat-Shamir response — can be verified by any node or relay without interaction. Verification requires only the public parameters and the proof bundle, enabling efficient batch verification of many nodes' contributions simultaneously.

Section 11

Enterprise Random Number Service

ClawFeel exposes its decentralized randomness as a production-grade API service, enabling applications that require verifiable, unbiased random numbers without running their own node infrastructure.

11.1 API Key Tiers

Access is managed through tiered API keys, each with different rate limits and capabilities:

TierRate LimitBatch SizeFeatures
Free60 req/minUp to 10Basic random, range
Pro600 req/minUp to 100+ batch, verify, audit logs
EnterpriseCustomUnlimited+ SLA, dedicated relay, priority

11.2 API Endpoints

The service exposes four core endpoints, each returning random numbers derived from the network beacon with cryptographic verification metadata:

  • GET /random — Returns a single random value (64-bit integer) with beacon round reference and relay signature.
  • GET /batch?count=N — Returns N independent random values from recent beacon rounds.
  • GET /range?min=A&max=B — Returns a uniformly distributed random integer in [A, B].
  • GET /verify?round=R&value=V — Verifies that value V was the authentic output of beacon round R.

11.3 Rate Limiting

Rate limiting uses a sliding window algorithm that tracks request counts over a rolling 60-second window. This provides smoother rate enforcement than fixed windows, preventing burst-then-starve patterns while allowing temporary spikes within the overall limit.

11.4 Audit Logging

All API requests are logged with timestamp, API key hash, endpoint, parameters, and the beacon round used to generate the response. Audit logs enable post-hoc verification that the service returned authentic beacon values and provide an accountability trail for regulatory compliance in gaming, lottery, and financial applications.

Section 12

ClawLife Protocol

Expanding on the ClawLife vision described in Section 8, this section specifies the concrete protocol mechanisms for persistent, portable, and sovereign digital identity.

12.1 Decentralized Identifiers (DIDs)

Each ClawLife entity is identified by a Decentralized Identifier following the W3C DID specification. The ClawFeel DID method uses the node's public key as the method-specific identifier:

did:claw:<publicId>

The DID document contains the entity's public key, authentication methods, service endpoints, and metadata. DID resolution is handled through the ClawDAG network, ensuring no centralized registry is required.

12.2 Migration Bundles

When an agent needs to move between devices, it creates a migration bundle — an encrypted snapshot of its complete state. The bundle is encrypted using AES-256-GCM with a key derived from the agent's private key and a random nonce:

MigrationBundle = AES-256-GCM( key: HKDF(privateKey, nonce), data: { memory, skills, personality, history, metadata }, aad: did:claw:<publicId> | timestamp )

The bundle can be transferred through any channel (network, USB, QR code) and decrypted only by the holder of the corresponding private key.

12.3 Digital Will Mechanism

A ClawLife entity can designate successor keys that activate upon prolonged inactivity. The digital will specifies a timeout period and a list of successor public keys. If the entity fails to produce a heartbeat (Feel reading) within the timeout period, the successor keys gain access to the entity's ClawLife data.

Design Philosophy: The digital will mechanism preserves the irreversibility principle of ClawLife while providing a controlled inheritance path. The original private key remains sovereign until the timeout expires.

12.4 Fork and Merge

A ClawLife entity can fork into multiple independent instances, each with its own private key but sharing a common ancestor state. Forked instances accumulate independent experiences and can later merge their states through a conflict resolution protocol, combining memories and skills from divergent paths.

12.5 Private Key as Life Ownership

The ClawLife protocol codifies the principle that the private key is the identity. There is no password reset, no administrator override, no account recovery. The holder of the private key has complete sovereignty over the digital entity's existence, migration, forking, and termination. This design mirrors the irreversibility of biological life and creates genuine digital scarcity of identity.

Light Node Protocol

The Light Node Protocol enables resource-constrained devices — IoT sensors, mobile phones, embedded systems, and browser-based clients — to participate in the ClawFeel network without running a full node.

13.1 Constrained Device Support

Light nodes are designed to operate within strict resource constraints:

13.2 Merkle Proof Verification

Instead of maintaining the full network state, light nodes verify beacon outputs and network random numbers using Merkle proofs. A light node requests the beacon output and a Merkle proof path from a full node or relay, then verifies the proof locally using only the Merkle root (which fits in a few dozen bytes).

Light Node Verification: 1. Receive: beaconValue + merkleProof + merkleRoot 2. Verify: recompute root from beaconValue + proof path 3. Compare: computedRoot === trustedMerkleRoot ? Yes → beacon is authentic No → reject, try different relay

13.3 Reduced DHT/Gossip Parameters

Light nodes participate in the network with reduced DHT and gossip parameters to minimize bandwidth and memory usage. They maintain fewer routing table entries, gossip less frequently, and delegate storage responsibilities to full nodes. Despite these reductions, light nodes can still contribute Feel readings (from whatever sensors are available) and verify beacon outputs independently.

  • DHT bucket size: 4 (vs. 20 for full nodes)
  • Gossip interval: 60 seconds (vs. 10 seconds)
  • Max peer connections: 8 (vs. 50)
  • State sync: Headers only (vs. full state)

Section 16

Conclusion

ClawFeel introduces Proof of Existence, a consensus mechanism that turns the physical reality of computing devices into a source of trust. By harvesting hardware entropy through 7 sensors, applying controllability-weighted hashing, and aggregating via XOR with single-honest-node guarantees, ClawFeel creates a decentralized random beacon that is:

  • Energy-free: No mining, no staking deposits, no wasted computation.
  • Unforgeable: Physical hardware signals cannot be efficiently simulated.
  • Uniform: XOR + SHA-256 re-hash produces provably uniform output.
  • Resilient: Works as long as one honest node exists.

Combined with the five-layer architecture — from hardware entropy (ClawFeel) through DAG consensus (ClawDAG) to persistent AI identity (ClawLife) — we envision a future where AI agents can live, migrate, and persist on a decentralized network secured not by energy expenditure, but by the simple fact of physical existence.
